# TestFlying 应用构建单配置模型设计

## 背景

当前 TestFlying 按 `development` 和 `production` 为同一个应用保存两份构建配置。两份配置都包含
Git 仓库、仓库子目录、Runner labels、凭据引用、产物类型和默认参数。实际业务中，这些内容描述的是
同一个应用如何被构建，不属于环境本身；开发环境和线上环境只需要在发起构建时选择，并传给 Runner
和 package-agent。

现有模型会产生以下问题：

- 同一个仓库和凭据需要重复填写。
- 两份配置可能逐渐不一致，无法判断哪一份代表应用的真实构建入口。
- 浏览器在触发构建时重复提交完整配置，可能用过期页面数据覆盖服务端状态。
- “接入应用”和“编辑构建配置”都需要先选择环境，增加无业务价值的步骤。

本设计采用真正的单配置模型。它覆盖
`2026-07-13-admin-build-settings-navigation-design.md` 中“不改变应用级开发环境、线上环境构建配置的
数据模型”以及“按环境配置应用”的相关约束。构建环境仍然存在，但只属于一次构建任务。

## 目标

1. 每个应用最多保存一份共享构建配置。
2. 开发环境和线上环境只在发起构建时选择。
3. Runner 和 package-agent 继续收到明确的构建环境。
4. 触发构建时由服务端读取共享配置并生成不可变快照。
5. 将已有双配置数据无损、确定性地合并到单配置模型。
6. 删除管理后台内部的旧双配置接口和前端状态，不保留两套行为。

## 非目标

- 不提供环境级字段覆盖。
- 不改变 `Build.environment`、`Build.requested_environment` 的语义。
- 不改变 Runner 轮询、任务领取、日志、制品上传和 package-agent 协议。
- 不改变应用平台、商店环境或分发环境的定义。
- 不新增构建环境，仍只允许 `development` 和 `production`。

## 领域模型

### 应用构建配置

`AppBuildSetting` 改为应用的一对一配置：

- `app_id`：唯一且非空。
- `git_url`：Git 仓库地址。
- `repo_subpath`：仓库内应用目录。
- `runner_labels_json`：节点匹配标签。
- `credential_refs_json`：凭据引用。
- `artifact_type`：预期产物类型。
- `optional_defaults_json`：可选默认构建参数。
- `created_at`、`updated_at`：审计时间。

删除 `environment` 字段以及 `(app_id, environment)` 唯一约束，改为 `app_id` 唯一约束。

### 构建任务

`Build` 继续保存：

- `environment`：实际执行环境。
- `requested_environment`：用户请求的环境。
- `runner_labels_json`：创建任务时生成的完整不可变构建快照。

构建快照继续包含仓库、子目录、凭据、产物类型、默认参数和环境。共享配置后续被编辑时，不得改变
已经创建的构建任务。

## 数据迁移

迁移按应用分组处理已有 `app_build_settings`：

1. 应用只有一条记录时直接保留其配置。
2. 应用有多条记录时，保留 `updated_at` 最新的一条。
3. `updated_at` 相同时优先保留 `development`，再按稳定主键排序，保证 PostgreSQL 和 SQLite 结果一致。
4. 删除未保留的记录。
5. 删除 `environment` 字段和旧唯一约束，增加 `app_id` 唯一约束。

该规则优先保留用户最后修改的意图，不尝试按字段合并两份配置，避免生成从未被用户实际保存过的组合。
迁移必须同时支持 PostgreSQL 和测试使用的 SQLite。

## 服务端行为

### 读取应用构建状态

应用详情和构建工作区统一返回一个 `buildSetting`，不再返回按环境分组的 `settings` 或
`environments` 数组。共享状态至少包含：

- 是否已配置。
- Git 仓库和子目录。
- Runner labels 和当前可匹配在线节点数量。
- 凭据引用名称、产物类型和默认参数。
- 最近更新时间。

未配置应用仍可出现在“尚未接入”的已有应用列表中。

### 保存配置

使用单数资源接口：

```text
PUT /admin/api/apps/{appId}/build-setting
```

请求体只包含共享构建配置字段，不包含环境。保存采用 upsert 语义：不存在时创建，存在时更新。

服务端继续执行现有校验：

- Git URL、产物类型等必填字段不得为空。
- `repo_subpath` 不得越过仓库根目录。
- Runner labels 和凭据引用必须是合法、去重后的字符串集合。
- 默认参数必须符合既有 JSON 结构约束。

旧的 `PUT /admin/api/apps/{appId}/build-settings/{environment}` 管理接口直接删除，不提供兼容别名。

### 发起构建

继续使用：

```text
POST /admin/api/apps/{appId}/builds
```

请求体收敛为：

```json
{
  "environment": "development",
  "gitRef": "main"
}
```

其中 `environment` 只能是 `development` 或 `production`。服务端按以下顺序处理：

1. 校验应用存在。
2. 校验应用已经保存共享构建配置。
3. 校验环境和 Git ref。
4. 读取共享配置并生成构建快照。
5. 创建排队任务并返回构建 ID 和状态。

没有匹配在线节点时仍允许任务排队，并返回明确提示；未配置应用时拒绝创建构建，返回可直接展示的
管理后台错误。

## 前端交互

### 接入和编辑配置

“接入已有应用”和“编辑构建配置”共用同一个配置弹窗：

- 不再展示开发环境、线上环境切换控件。
- 只展示一份仓库、子目录、节点标签、凭据、产物类型和默认参数表单。
- 保存成功后刷新该应用的共享配置摘要和节点匹配状态。
- 未保存配置前不能发起构建。

### 发起构建

应用构建操作区保留：

- 环境选择：开发环境或线上环境。
- Git ref 输入。
- 共享配置只读摘要。
- 当前匹配节点状态。
- “立即构建”按钮。

环境选择不再影响配置表单内容。切换环境只改变本次构建请求及最终 Runner/package-agent 载荷。

### 状态与错误

- 保存配置和发起构建分别展示独立 loading 状态，禁止重复提交。
- 配置校验错误显示在对应表单字段或弹窗顶部。
- 构建创建失败保留用户当前选择和 Git ref，不关闭操作区。
- 构建创建成功不跳页，展示构建 ID、排队状态和进入构建记录的入口。

## Runner 与 package-agent 数据流

```text
应用共享构建配置
        +
构建时选择的 environment / gitRef
        |
        v
中心服务生成不可变 Build 快照
        |
        v
Runner 领取任务
        |
        v
package-agent 按 environment 执行开发或线上构建
```

Runner 节点继续同时支持 iOS 和 Android。节点匹配只依据应用平台能力和共享配置中的 labels，不按
开发环境或线上环境拆分节点配置。

## 测试与验收

### 后端

- 迁移一条、两条和更新时间相同的旧配置，验证合并结果稳定。
- 同一应用重复保存只更新同一条配置。
- 未配置应用不能发起构建。
- 同一共享配置可以分别创建开发环境和线上环境构建。
- 两种构建的快照配置相同，但环境字段不同。
- 修改共享配置不会改变已有构建快照。
- 非法环境、Git ref、仓库子目录和凭据引用返回结构化管理错误。

### 前端

- 配置弹窗不再出现环境切换。
- 应用配置保存、重新打开和编辑流程正确。
- 构建环境选择仍可切换开发环境和线上环境。
- 发起构建请求不再携带共享配置字段。
- loading、错误和成功状态不引起整页跳转。

### 回归

- Runner 可领取两种环境的任务。
- package-agent 收到正确环境。
- 构建记录、日志、制品、安装链接和通知行为不变。
- PostgreSQL、SQLite、前端类型检查、前后端测试、Go 测试和生产构建全部通过。

## 完成标准

1. 数据库中一个应用最多存在一条构建配置。
2. 后台任何页面都不再要求分别配置开发环境和线上环境。
3. 用户仍能在发起构建时选择开发环境或线上环境。
4. Runner 与 package-agent 收到正确环境和完整不可变配置快照。
5. 旧配置迁移、管理 API、前端交互和回归测试全部通过。
